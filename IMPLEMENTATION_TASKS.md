# Задачи реализации AI-сервиса подписи PDF

Документ основан на `SIGNING_APP_SPEC.md` и описывает, что именно нужно сделать для первой рабочей версии приложения.

Приоритеты:

- `P0` - обязательно для минимально рабочей версии.
- `P1` - нужно для качественной первой версии.
- `P2` - улучшения после стабильного MVP.

## 1. Подготовка проекта

### P0. Инициализировать backend-проект

Задачи:

- Создать структуру Python-приложения.
- Добавить зависимости для web API, PDF-обработки, изображений, OCR/CV, OpenAI API и тестов.
- Добавить `.env.example` с переменными:
  - `OPENAI_API_KEY`
  - `OPENAI_BASE_URL=https://api.openai.com/v1`
  - `OPENAI_MODEL`
  - `SIGNATURE_IMAGE_PATH=/Users/rafailvv/Documents/подпись.png`
  - `STAMP_IMAGE_PATH=/Users/rafailvv/Documents/Печать.png`
  - `WORKDIR`
- Добавить базовую конфигурацию приложения.
- Добавить команду запуска локального сервера.

Критерии готовности:

- Сервер запускается локально.
- Есть healthcheck endpoint.
- Конфигурация читается из env.
- Приложение не падает, если OpenAI key не задан.

### P0. Определить структуру данных

Задачи:

- Реализовать модели:
  - `DocumentJob`
  - `PageAnalysis`
  - `SignatureTarget`
  - `AIPlacementDecision`
  - `Placement`
  - `ExportResult`
- Зафиксировать единый формат координат: PDF points.
- Добавить конвертацию между preview pixels и PDF points.
- Добавить статусы задач:
  - `uploaded`
  - `analyzing`
  - `needs_review`
  - `ready`
  - `exported`
  - `failed`

Критерии готовности:

- Модели валидируются.
- Ошибочные координаты и пустые обязательные поля отклоняются.
- Координаты сохраняются независимо от масштаба предпросмотра.

## 2. Backend API

### P0. Реализовать загрузку PDF

Endpoint: `POST /upload`

Задачи:

- Принимать несколько PDF за один запрос.
- Проверять MIME/type и расширение.
- Сохранять исходники во временную рабочую папку.
- Создавать `DocumentJob` для каждого файла.
- Возвращать список `job_id`.

Критерии готовности:

- Можно загрузить несколько PDF.
- Не-PDF файлы отклоняются.
- Ошибка одного файла не ломает весь пакет.
- Исходный файл не изменяется.

### P0. Реализовать запуск анализа

Endpoint: `POST /analyze`

Задачи:

- Принимать список `job_id`.
- Принимать опции:
  - `place_signature`
  - `place_stamp`
  - `add_name_if_missing`
  - `use_ai`
  - `require_manual_confirmation`
- Для каждого документа запускать PDF-анализ.
- Возвращать placements, warnings и статусы.

Критерии готовности:

- Анализ работает для одного и нескольких документов.
- Опции сохраняются в `DocumentJob`.
- Документ с ошибкой получает статус `failed`, остальные продолжают обработку.

### P0. Реализовать preview API

Endpoint: `GET /preview/{job_id}`

Задачи:

- Возвращать список страниц.
- Возвращать размеры страниц в PDF points.
- Возвращать URL preview-изображений.
- Возвращать текущие placements.
- Возвращать warnings/confidence/reason.

Критерии готовности:

- Frontend может показать страницу и overlay.
- Координаты overlay совпадают с PDF-координатами после масштабирования.

### P0. Реализовать сохранение ручных правок

Endpoint: `POST /placement/{job_id}`

Задачи:

- Принимать обновленные placements.
- Поддержать изменение координат, размеров, включения/выключения подписи, печати и ФИО.
- Поддержать добавление и удаление placements.
- Помечать документ как подтвержденный пользователем.

Критерии готовности:

- Ручная правка сохраняется.
- Экспорт использует именно ручные coordinates.
- Удаленные placements не экспортируются.

### P0. Реализовать экспорт

Endpoints:

- `POST /export`
- `GET /download/{export_id}`

Задачи:

- Генерировать PDF с наложенными подписью, печатью и ФИО.
- Сохранять PNG-прозрачность.
- Генерировать отдельные PDF.
- Генерировать ZIP для нескольких документов.
- Не перезаписывать исходники.
- Блокировать экспорт, если включено ручное подтверждение и документ не подтвержден.

Критерии готовности:

- PDF открывается в Preview/Adobe Reader.
- Визуальный результат соответствует предпросмотру.
- ZIP содержит все успешно обработанные документы.

## 3. PDF-анализ и локальный алгоритм

### P0. Читать PDF и получать геометрию страниц

Задачи:

- Открывать PDF.
- Получать количество страниц.
- Получать размеры страниц.
- Сохранять page metadata в `PageAnalysis`.

Критерии готовности:

- Корректно обрабатываются одно- и многостраничные PDF.
- Ошибочные PDF дают понятную ошибку.

### P0. Извлекать текстовый слой

Задачи:

- Получать текст страницы.
- Получать координаты слов.
- Нормализовать текст для поиска.
- Сохранять исходный текст для UI/debug.

Критерии готовности:

- На PDF с текстовым слоем находятся слова и координаты.
- Координаты слов соответствуют PDF page coordinates.

### P1. Добавить OCR fallback

Задачи:

- Определять, что текстовый слой отсутствует или плохой.
- Рендерить страницу для OCR.
- Распознавать русский текст.
- Получать bounding boxes слов.
- Приводить OCR-координаты к PDF points.

Критерии готовности:

- Сканированный PDF без text layer анализируется.
- При плохом OCR документ помечается как `needs_review`.

### P0. Искать якорные слова и фразы

Задачи:

- Искать точные и нормализованные совпадения:
  - `Венедиктов`
  - `Венедиктов Р.В.`
  - `Венедиктов Р В`
  - `Генеральный директор`
  - `директор`
  - `подпись`
  - `ФИО`
  - `расшифровка подписи`
- Добавить fuzzy matching для OCR-ошибок.
- Понижать confidence, если найденная фамилия находится в обычном тексте, а не в блоке подписания.

Критерии готовности:

- На типовом документе находится зона рядом с `Венедиктов Р.В.`.
- Упоминание `Венедиктов Р.В.` в середине договора не считается подписью без линии/контекста.

### P0. Искать горизонтальные линии

Задачи:

- Извлекать векторные линии из PDF, если доступны.
- На preview/rendered изображении искать горизонтальные линии через CV.
- Фильтровать слишком короткие, слишком длинные и табличные линии.
- Связывать линии с ближайшими якорями.

Критерии готовности:

- Находятся линии подписи.
- Табличные линии получают низкий confidence.

### P0. Формировать `SignatureTarget`

Задачи:

- Объединять найденные якоря и линии.
- Рассчитывать локальный confidence.
- Сохранять reason:
  - `line_near_venediktov`
  - `line_near_general_director`
  - `empty_signature_line`
  - `stamp_area_near_signature`
  - `ambiguous_multiple_lines`
  - `low_text_confidence`
- Помечать неоднозначные случаи как `needs_manual_review`.

Критерии готовности:

- Для каждого кандидата есть bbox, reason и confidence.
- Нет автоподписи при confidence ниже минимального порога.

## 4. AI-анализ

### P0. Добавить OpenAI-клиент

Задачи:

- Читать `OPENAI_API_KEY`, `OPENAI_BASE_URL`, `OPENAI_MODEL`.
- Не делать AI-запросы, если `OPENAI_API_KEY` отсутствует.
- Поддержать timeout и retry.
- Логировать техническую ошибку без записи секретов.

Критерии готовности:

- Приложение работает без OpenAI key.
- При ошибке API документ не теряется и не падает весь анализ.

### P0. Подготовить AI payload

Задачи:

- Передавать OCR/text-layer текст страницы.
- Передавать координаты слов.
- Передавать линии.
- Передавать кандидаты.
- При необходимости передавать crop-изображения вокруг кандидатов.
- Не отправлять весь PDF без необходимости.

Критерии готовности:

- Payload содержит достаточно контекста для выбора места.
- Payload не содержит лишних страниц и областей.

### P0. Реализовать structured AI response

Задачи:

- Сформировать prompt с инструкцией вернуть только JSON.
- Описать случаи, которые AI должен различать:
  - ФИО рядом с зоной подписи;
  - ФИО в обычном тексте;
  - только `Генеральный директор`;
  - несколько мест подписи;
  - подпись и печать в разных местах;
  - документ уже подписан;
  - надежного места нет.
- Валидировать JSON.
- Преобразовывать ответ в `AIPlacementDecision`.

Критерии готовности:

- Невалидный JSON не ломает анализ.
- AI может вернуть несколько decisions.
- Низкая уверенность AI ведет к `needs_manual_review`.

### P1. Реализовать AI fallback policy

Задачи:

- При ошибке AI использовать локальные candidates.
- Если локальный confidence низкий, требовать ручной проверки.
- В UI показывать warning: AI недоступен или ответ невалиден.

Критерии готовности:

- Сервис остается работоспособным без AI.
- Пользователь видит, почему документ требует проверки.

## 5. Расчет размещения

### P0. Рассчитать подпись

Задачи:

- Загружать PNG подписи.
- Сохранять пропорции.
- Рассчитывать bbox по линии подписи.
- Масштабировать до 60-90% ширины линии, если линия надежная.
- Для нестандартных линий использовать типовой физический размер и ставить `needs_manual_review`.

Критерии готовности:

- Подпись выглядит реалистично.
- Подпись не растягивается непропорционально.
- Подпись не закрывает основной текст.

### P0. Рассчитать печать

Задачи:

- Загружать PNG печати.
- Сохранять пропорции и прозрачность.
- Задавать физический размер 35-45 мм.
- Ставить печать рядом с подписью.
- Делать частичное перекрытие подписи на 15-35%.
- Если найдена отдельная зона `М.П.` или `печать`, разрешить отдельное размещение.

Критерии готовности:

- Печать частично захватывает подпись.
- Печать не закрывает ФИО полностью.
- Размер печати выглядит естественно на A4.

### P1. Рассчитать вставку ФИО

Задачи:

- Проверять, есть ли `Венедиктов Р.В.` рядом с местом подписи.
- Если есть только `Венедиктов`, не дублировать фамилию без необходимости.
- Вставлять `Венедиктов Р.В.` при включенной опции.
- Подбирать размер шрифта по окружающему тексту.

Критерии готовности:

- ФИО не дублируется.
- Вставленный текст визуально близок к документу.

## 10. Минимальный MVP

MVP считается готовым, когда работает такой flow:

1. Пользователь загружает несколько PDF.
2. Выбирает `Поставить подпись` и/или `Поставить печать`.
3. Сервис находит типовую зону рядом с `Венедиктов Р.В.` или `Генеральный директор`.
4. Пользователь видит предпросмотр.
5. Пользователь может поправить подпись и печать вручную.
6. Сервис экспортирует PDF.
7. Для нескольких документов сервис экспортирует ZIP.

AI в MVP может быть включен после стабильного локального pipeline, но архитектура должна быть подготовлена под AI с самого начала: модели данных, `use_ai`, warnings и fallback должны существовать сразу.

## 11. Что не входит в первую версию

- DOCX.
- Юридическая электронная подпись/ЭЦП.
- Облачное хранение документов.
- Публичный SaaS и авторизация пользователей.
- Автоматическое обучение на пользовательских документах.
- Полностью автоматический экспорт без возможности ручной проверки.
